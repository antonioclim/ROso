The find Command - Introduction

find searches for files in the directory hierarchy based on various criteria.

General Syntax

find [cale] [expresii] [ac»õiuni]

Components

  • path - where to search (default: .)
  • expressions - filtering criteria
  • actions - what to do with the results

Simple Example

# Find all .txt files

üí° Why find and not ls?
find can search recursively, filter by size, date, permissions and can execute commands on the results!

find - Basic Tests

Search by Name

# Exact match (case-sensitive)
find . -name "config.txt"

# With pattern (globbing)
find /home -name "*.log"

# Case-insensitive
find . -iname "README*"

Search by Type

Flag Type Example
-type f Regular file find . -type f
-type d Directory find . -type d
-type l Symbolic link find /usr -type l

find - Size and Time

Search by Size

# Files larger than 100MB
find /var -size +100M

# Files smaller than 1KB
find . -size -1k

# Files exactly 0 bytes (empty)
find . -size 0 -type f

Suffixes: c=bytes, k=KB, M=MB, G=GB

Search by Modification Time

# Modified in the last 24 hours
find . -mtime -1

# Modified more than 30 days ago
find /var/log -mtime +30

# Modified in the last 60 minutes
find . -mmin -60
üìå Interpreting -mtime
-mtime -N = last N days | -mtime +N = older than N days | -mtime N = exactly N days

find - Logical Operators

AND (implicit)

# .txt files larger than 1MB
find . -type f -name "*.txt" -size +1M

OR (-o)

# .txt OR .log files
find . -name "*.txt" -o -name "*.log"

NOT and Grouping

# Everything EXCEPT .git
find . ! -name ".git"

# Grouping with parentheses (escaped!)
find . -type f \( -name "*.txt" -o -name "*.md" \)

# Complex: large .log files older than 7 days
find /var/log -type f -name "*.log" -size +10M -mtime +7
‚ö†Ô∏è Watch the parentheses!
Parentheses must be escaped with \( »ôi \) or put in quotes.

find - Actions (-exec)

Execution for Each File

# Displays details for each file found
find . -name "*.txt" -exec ls -l {} \;

# {} = placeholder for the current file
# \; = terminates the command (escaped ;)

Batch Execution (more efficient)

# Sends ALL files as arguments at once
find . -name "*.txt" -exec wc -l {} +
-exec cmd {} \; ‚Üí cmd file1; cmd file2; cmd file3; (3 procese) -exec cmd {} + ‚Üí cmd file1 file2 file3; (1 proces)

With Confirmation (-ok)

# Asks for confirmation before each execution
find . -name "*.tmp" -ok rm {} \;

find -delete - DANGER!

üö® CRITICAL WARNING!
The order of options in find matters! -delete placed incorrectly can delete EVERYTHING!

WRONG - Deletes EVERYTHING!

# -delete BEFORE -name
find . -delete -name "*.tmp"
# Disaster! Deletes everything from .

CORRECT

# -name BEFORE -delete
find . -name "*.tmp" -delete
# Deletes only .tmp

Best Practice: Test First!

# Step 1: See what will be deleted
find . -name "*.tmp" -print

# Step 2: If OK, delete
find . -name "*.tmp" -delete

$@ vs $* - The Crucial Difference

üéØ Common Misconception
Many believe that $@ and $* are identical. THEY ARE NOT! 
#!/bin/bash
# Run: ./test.sh "hello world" test

echo "=== With \$@ ==="
for arg in "$@"; do echo "[$arg]"; done
# Output:
# [hello world]
# [test]

echo "=== With \$* ==="
for arg in "$*"; do echo "[$arg]"; done
# Output:
# [hello world test]
"$@" = "arg1" "arg2" "arg3" ‚Üí 3 elemente separate "$*" = "arg1 arg2 arg3" ‚Üí 1 string concatenat
üí° Rule
Use always "$@" (with quotes!) to preserve arguments with spaces.

shift - Shifting Arguments

shift removes the first argument and shifts the rest to the left.

#!/bin/bash
# Run: ./script.sh a b c d

echo "Before: \$1=$1 \$2=$2 \$3=$3"  # a b c
shift
echo "After shift: \$1=$1 \$2=$2 \$3=$3"  # b c d
shift 2
echo "After shift 2: \$1=$1"  # d

Pattern: Processing All Arguments

while [[ $# -gt 0 ]]; do
    echo "Processing: $1"
    shift
done
üìå Usage
shift is useful when you have options at the beginning and files at the end - you process the options, then $@ contains only the files.

getopts - Option Parsing

getopts parses options like -a, -b value in a standardised way.

Syntax

while getopts "optstring" opt; do
    case $opt in
        # handle options
    esac
done

Optstring Explained

Pattern Meaning Example
a Simple flag (boolean) -a
b: Option with mandatory argument -b value
:ab: : at beginning = silent errors manual error handling

getopts - Complete Example

#!/bin/bash
# Script with options: -h (help), -v (verbose), -o FILE (output)

verbose=0
output=""

while getopts "hvo:" opt; do
    case $opt in
        h)
            echo "Usage: $0 [-h] [-v] [-o file] args..."
            exit 0
            ;;
        v)
            verbose=1
            ;;
        o)
            output="$OPTARG"  # OPTARG = the argument value
            ;;
        \?)
            echo "Invalid option: -$OPTARG" >&2
            exit 1
            ;;
    esac
done

# Remove processed options, leave only arguments
shift $((OPTIND - 1))

# Now $@ contains only non-option arguments
echo "Files to process: $@"

OPTARG

Contains the argument value for options with :

# -o file.txt
$OPTARG = "file.txt"

OPTIND

Index of the next argument to process

# After getopts

./script.sh -v -o out.txt file1.txt file2.txt ‚Üë ‚Üë ‚Üë ‚Üë 1 2 3 OPTIND=4 dupƒÉ parsare DupƒÉ shift $((OPTIND-1)): $1 = file1.txt $2 = file2.txt
‚ö†Ô∏è getopts Limitation
getopts does NOT support long options (--verbose). For these, use manual parsing with case.

chmod - Symbolic Mode

Syntax: WHO OPERATOR PERMISSION

WHO

  • u = user (owner)
  • g = group
  • o = others
  • a = all

OPERATOR

  • - = remove
  • = = set exactly
# Add execute for owner

# Remove write for group and others

# Set exactly: owner=rwx, group=rx, others=r

# Add read for all

chmod Recursive: X vs x

‚ö†Ô∏è Common Problem
chmod -R a+x makes ALL files executable - including .txt, .jpg, etc.!

The Solution: Capital X

# ‚ùå WRONG: makes everything executable

# ‚úÖ CORRECT: X = execute doar pentru directoare
# and files that already have x for someone

Best Practice for Projects

# Set correct permissions for everything

find myproject/ -type f -exec chmod 644 {} \;

# Then make only scripts executable
find myproject/ -name "*.sh" -exec chmod 755 {} \;

umask - Default Permissions

üéØ Critical Misconception
umask does NOT set permissions - it REMOVES permissions from default!
Default permissions (without umask): Files: 666 (rw-rw-rw-) Directories: 777 (rwxrwxrwx) umask 022: Files: 666 - 022 = 644 (rw-r--r--) Directories: 777 - 022 = 755 (rwxr-xr-x) umask 077: Files: 666 - 077 = 600 (rw-------) Directories: 777 - 077 = 700 (rwx------) 
# Check current umask
umask         # Displays 0022
umask -S      # Displays simbolic: u=rwx,g=rx,o=rx

# Set umask for session
umask 077     # Very restrictive
touch secret.txt
ls -l secret.txt  # -rw------- (600)

SUID - Set User ID

When SUID is set, the program runs with the permissions of the owner, not of the one executing it.

ls -l /usr/bin/passwd
# -rwsr-xr-x 1 root root ... /usr/bin/passwd
#    ‚Üë
#    s √Æn loc de x = SUID activat

Why does it exist?

/usr/bin/passwd needs to modify /etc/shadow (owned by root). SUID allows normal users to change their password.

Setting SUID

chmod u+s program     # Symbolic
chmod 4755 program    # Octal (4 = SUID)
üö® SUID on Scripts
For security reasons, Linux ignores SUID on bash scripts! It only works on compiled binaries.

SGID and Sticky Bit

SGID on Directories

New files inherit the directory's group, not the user's group.

# Set SGID on shared directory
chmod g+s /shared/project/
chmod 2775 /shared/project/  # 2 = SGID

ls -ld /shared/project/
# drwxrwsr-x ... (s √Æn pozi»õia group x)

Sticky Bit on Directories

Only the file owner can delete the file (e.g.: /tmp).

ls -ld /tmp
# drwxrwxrwt ... (t √Æn pozi»õia others x)
#          ‚Üë
#          t = Sticky bit

# Setting
chmod +t /shared/temp/
chmod 1777 /shared/temp/  # 1 = Sticky

What is Cron?

cron is a daemon that executes scheduled commands automatically.

Components

  • crond - the daemon
  • crontab - the job table
  • /var/spool/cron/ - user crontabs
  • /etc/crontab - system crontab

Commands

# Edit your crontab
crontab -e

# List jobs
crontab -l

# Delete ALL jobs!
crontab -r
üö® WARNING!
crontab -r deletes ALL jobs without confirmation! Easily confused with -e (they are neighbours on the keyboard).

Cron - Common Pitfalls

üéØ Misconception #1: Execution environment
Cron runs with a minimal environment - it does NOT have access to your variables (customised PATH, HOME, etc.)! 
# ‚ùå WRONG - python may not be in PATH
* * * * * python script.py

# ‚úÖ CORRECT - calea completƒÉ
* * * * * /usr/bin/python3 /home/user/script.py
üéØ Misconception #2: DOM and DOW
If you specify BOTH day of month AND day of week, cron treats them with OR, not AND! 
# Runs on the 15th OR every Monday (not just Monday the 15th!)
0 9 15 * 1 /script.sh

Best Practice: Logging

# Redirect stdout and stderr to log
0 3 * * * /home/user/backup.sh >> /var/log/backup.log 2>&1
